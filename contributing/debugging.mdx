title: "Debugging WinUI Build Failures and Crashes"

When building WinUI from source or working with WinUI-based apps, you may encounter build failures or runtime crashes that require deeper investigation. This page covers the two main areas: reading build logs to diagnose compilation failures, and collecting crash dumps to find the root cause of runtime crashes.

## Debugging build failures

### Reading binlog files

Every WinUI build produces a `.binlog` file alongside the standard console output. Binlogs capture the full MSBuild execution in a structured, machine-readable format that is far easier to navigate than raw text output.

To view a binlog file in a human-readable way, use the [MSBuild Structure Log Viewer](http://msbuildlog.com/). You can install it directly from the root of the WinUI repo:

```cmd
.\OneTimeSetup.cmd -Install LogViewer
```

The viewer presents errors, warnings, targets, and tasks in a tree structure, making it straightforward to identify exactly which target failed and why.

### Collecting binlogs via the command line

The recommended approach for collecting binlogs is through the MSBuild CLI, as these are generally easier to diagnose.

<Note>
  When investigating build failures in the WinUI source code, run `devcmd.cmd` at the root of the repository before any MSBuild commands to initialize the correct environment.
</Note>

<CodeGroup>

```cmd Build solution (x86 Release)
msbuild /p:Platform=x86 /p:Configuration=Release /bl
```

```cmd Build app package
msbuild /p:AppxBundlePlatforms=x86 /p:Platform=x86 /p:Configuration=Release /p:BuildAppxUploadPackageForUap=true /bl
```

</CodeGroup>

These commands produce a `.binlog` file in the working directory. Share that file when reporting build issues.

### Collecting binlogs via Visual Studio

If you prefer to collect binlogs from within Visual Studio:

<Steps>
  <Step title="Install the Project System Tools extension">
    Download and install the extension for your Visual Studio version:
    - [Project System Tools for VS 2019](https://marketplace.visualstudio.com/items?itemName=VisualStudioProductTeam.ProjectSystemTools)
    - [Project System Tools for VS 2022](https://marketplace.visualstudio.com/items?itemName=VisualStudioProductTeam.ProjectSystemTools2022)
  </Step>
  <Step title="Set build log verbosity to Diagnostics">
    Go to **Tools → Options → Projects and Solutions → MSBuild project build log file verbosity** and set it to **Diagnostics**.
  </Step>
  <Step title="Open the Build Logging window">
    Go to **View → Other Windows → Build Logging**.
  </Step>
  <Step title="Start logging and reproduce the failure">
    Press the play button in the Build Logging window, then run the build steps that produce errors. Failed steps are marked as **Failed** and their `.binlog` files can be shared for debugging.
  </Step>
</Steps>

### Memory errors: C1076 and C1060

Building `Microsoft.UI.Xaml.Controls.vcxproj` occasionally produces these errors:

```
error C1076: compiler limit: internal heap limit reached
error C1060: compiler is out of heap space
```

These occur because the compiler runs out of heap memory during a parallel build. There are two ways to resolve them:

**Option 1 — Retry the build.** Run `build.cmd` again. An incremental build often succeeds on the second or third attempt, though it may require multiple retries.

**Option 2 — Limit parallel compiler processes.** This is the more reliable fix. Set the `CL` environment variable to cap the number of parallel compiler instances before building:

<CodeGroup>

```cmd CMD
set CL=/MP2
build.cmd
```

```powershell PowerShell
$env:CL = "/MP2"
.\build.cmd
```

</CodeGroup>

<Tip>
  `/MP2` allows up to 2 parallel compiler processes. If you continue to see errors, try `/MP1` to disable parallelism entirely.
</Tip>

## Debugging crashes

### Getting a crash dump with Visual Studio

If you can launch the crashing app from Visual Studio or attach before the crash occurs:

<Steps>
  <Step title="Configure the debugger">
    If launching from Visual Studio, set the debugger type to **Native Only** or **Mixed (Managed and Native)** before starting.

    If attaching to a running process, open **Attach to Process**, find the process, and ensure the **Attach to** setting includes **Native**.
  </Step>
  <Step title="Reproduce the crash">
    Run through the steps that trigger the crash. Visual Studio breaks in when the crash occurs.
  </Step>
  <Step title="Save the dump">
    From the **Debug** menu, select **Save Dump As...** to save a `.dmp` file.
  </Step>
</Steps>

<Warning>
  Visual Studio's Diagnostic Tools can occasionally cause crashes that only occur during a debugging session. If you see a crash with a function name containing `Diagnostics` on the stack, or a crash in a `ScriptedSandbox64.exe` process, try disabling **Tools → Options → Debugging → Enable Diagnostic Tools while debugging** and reproduce the crash again.
</Warning>

### Getting a crash dump with WinDbg

[WinDbg](https://docs.microsoft.com/en-us/windows-hardware/drivers/debugger/debugger-download-tools) gives you more control over dump collection. Once you have attached to the crashing process and the crash occurs, save a full memory dump with:

```windbg
.dump /ma <filename.dmp>
```

- If the crash happens after launch, attach to the running app first, then reproduce.
- If the crash happens on launch, use [WinDbg Preview](https://www.microsoft.com/store/p/windbg/9pgjgd53tn86), which includes a **Launch app package** option for UWP and packaged apps.

### Automated crash dump collection

For a fully automated approach, use the crash analysis script:

<Steps>
  <Step title="Download the script">
    Download the script to a local folder, for example `C:\temp`:
    ```
    https://aka.ms/RNW/analyze-crash.ps1
    ```
  </Step>
  <Step title="Allow script execution">
    Open an admin PowerShell and, if you haven't already, allow unsigned scripts:
    ```powershell
    Set-ExecutionPolicy Unrestricted
    ```
  </Step>
  <Step title="Run the script">
    Pass the name of your app's executable:
    ```powershell
    C:\temp\analyze-crash.ps1 -ExeName MyApp -Repo microsoft/microsoft-ui-xaml
    ```
    The script sets up automatic crash dump collection, downloads the native debugging tools (including `cdb`), and waits for you to reproduce the crash.
  </Step>
  <Step title="Reproduce and collect">
    Launch your app and trigger the crash. When it crashes, a `.dmp` file is generated. Press Enter to resume the script.
  </Step>
  <Step title="Review the analysis">
    The script runs `cdb` to analyze the dump, writes results to `analyze.log`, copies the log to the clipboard, and opens a browser to file an issue in the WinUI repo where you can paste the stack trace.
  </Step>
</Steps>

### Using Windows Error Reporting for local dumps

You can configure Windows to save crash dumps locally via registry keys. Run these commands in an administrative command prompt:

```cmd
reg add "HKLM\SOFTWARE\Microsoft\Windows\Windows Error Reporting\LocalDumps" /v DumpFolder /d "C:\dumps"
reg add "HKLM\SOFTWARE\Microsoft\Windows\Windows Error Reporting\LocalDumps" /v DumpType /t REG_DWORD /d 2
```

After reproducing the crash, find the `.dmp` file in `C:\dumps`.

### Reading crash callstacks

For most crashes (such as access violations), the direct crash callstack is the right starting point.

#### Stowed exception crashes (0xc000027b)

Stowed exception crashes are different — the actual error is saved earlier and the stack has usually unwound by the time the crash is declared fatal. Simply reading the direct callstack may not point to the real cause.

To get the correct callstack for a stowed exception crash:

<Steps>
  <Step title="Open the dump in WinDbg">
    Load the `.dmp` file in WinDbg.
  </Step>
  <Step title="Install the PDE extension">
    Download the PDE extension from the [OneDrive link](https://onedrive.live.com/?authkey=%21AJeSzeiu8SQ7T4w&id=DAE128BD454CF957%217152&cid=DAE128BD454CF957) (linked from the [Channel 9 stowed exceptions talk](https://channel9.msdn.com/Shows/Inside/C000027B)). Place the appropriate x64 or x86 `.dll` from the `PDE*.zip` into the `winext` directory of your WinDbg installation.
  </Step>
  <Step title="Dump the stowed exceptions">
    Run the following command in WinDbg:
    ```windbg
    !pde.dse
    ```
    This outputs all stowed exceptions with their associated HRESULT codes.
  </Step>
  <Step title="Identify the relevant exception">
    There are often multiple stowed exceptions. The first one is usually the most interesting. If the first exception appears to re-throw the second, and the second stack goes deeper into the same code path, the second exception may be the true origin of the error.
  </Step>
</Steps>

<Info>
  The HRESULT shown alongside each stowed exception is valuable context — it identifies the error code associated with that particular exception and can narrow down which subsystem is responsible.
</Info>
